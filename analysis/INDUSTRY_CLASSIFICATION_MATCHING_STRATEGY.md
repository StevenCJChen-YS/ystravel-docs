# Industry Classification Matching Strategy

## 1. 背景

目前已確認後續需要把篩選後的旅客 / 訂單，區分成：

- `同業`
- `直客`

目前推定依據會來自另一份 PDF 名單，該 PDF 初步檢視為：

- 同業公司基本資料表
- 內容以公司名稱為主
- 名稱多半是正式公司全名，例如 `某某旅行社股份有限公司`

而 Excel `來源帳號` 可能出現：

- 完整公司名
- 簡稱
- 公司名加聯絡人
- 公司名加分公司
- 平台名或通路名

因此不能只靠單純「完全相同字串」比對。

## 2. 目前看到的實際風險

Excel `來源帳號` 已出現這類型：

- `永信旅行社股份有限公司`
- `永信旅行社`
- `全聯旅行社-何思霈`
- `威順旅行社-田慈慧`
- `旅遊經理人-陳涬燕`
- `易遊網-杜文馨`

這代表同一家公司在 Excel 內部就可能已經有多種寫法。

## 3. 建議的比對層級

### Level 1: Raw Exact

Excel `來源帳號` 與 PDF 公司名稱完全相同。

範例：

- `永信旅行社股份有限公司` = `永信旅行社股份有限公司`

這種可以直接自動判定為 `同業`。

### Level 2: Normalized Exact

先做名稱正規化後再比對。

正規化包含：

- 去掉 `股份有限公司`
- 去掉 `有限公司`
- 去掉 `(股)` / `(股)公司`
- 去掉 `台北分公司 / 台中分公司 / 高雄分公司 ...`
- 去掉尾端聯絡人，例如 `-何思霈`
- 去掉多餘空白與符號

範例：

- Excel: `全聯旅行社-何思霈`
- PDF: `全聯旅行社股份有限公司`

正規化後都可落到 `全聯旅行社`

這類可列為 `高信心自動命中`。

### Level 3: Root / Alias Candidate

再進一步拿掉常見產業字尾後做候選比對，例如：

- 去掉 `旅行社`
- 去掉 `國際`
- 去掉 `旅遊`
- 去掉 `觀光`

範例：

- Excel: `首都`
- PDF: `首都旅行社`

這種不建議直接自動判定同業，而應列為：

- `candidate`
- 進人工覆核

原因是這種短字根容易撞名，風險較高。

## 4. 正式落地建議

之後實作 `同業 / 直客` 分類時，建議分 3 層輸出：

### 4.1 auto_match

符合以下任一：

- raw exact
- normalized exact

可直接判為 `同業`

### 4.2 review_required

符合以下狀況：

- 只有短根字相同
- 互相包含但不完全一致
- 名稱過短，例如只有 2 到 3 個字
- Excel 有聯絡人尾碼但基底名稱不夠明確

這批不直接判同業，進人工確認。

### 4.3 no_match

無法跟 PDF 同業名單建立合理關聯。

這批可先列為 `直客候選`，但仍保留覆核欄位。

## 5. 目前不建議的做法

- 不建議用單純 `contains` 就直接判同業
- 不建議只因為有共同兩個字就自動命中
- 不建議把平台名、聯絡人名、部門名直接當公司主名稱

## 6. 後續待辦

1. 解析 PDF 全部公司名稱成結構化名單
2. 將 Excel `來源帳號` 做 alias normalization
3. 建立 `auto_match / review_required / no_match` 對照結果
4. 跟你確認幾個高風險縮寫是否可接受自動判定
5. 最後再把同業 / 直客邏輯寫進正式資料匯入流程
