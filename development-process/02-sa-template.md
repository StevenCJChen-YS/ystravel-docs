# SA(系統分析)模板

> 複製到 `docs/features/<feature-slug>/sa.md`。**這份由 AI 草擬,Steven 負責看懂重點並確認方向,不要求 Steven 自己寫技術細節。** 如果 Steven 看不懂某一段,請 AI 用白話重寫,不要放著看不懂的內容直接簽核。

```markdown
# SA: <功能名稱>

| 項目 | 內容 |
|---|---|
| 對應 PRD | ./prd.md |
| 狀態 | 草稿 / 已確認 |
| 建立日期 | YYYY-MM-DD |

## 0. 白話摘要

3-5 句話：這份在分析什麼、關鍵結論、最大的風險或取捨。讓不讀技術細節的人看完這段就掌握重點。下面各節是專業展開。

## 1. 系統邊界

這個功能會動到哪些既有系統?哪些不會?

- 涉及系統:CRM Backend / CRM Frontend / AuthService / EIP / ...
- 是否已有相關架構文件?（優先引用,不要重複寫)
  - 例如 [CRM_AUTH_SERVICE_SA.md](../architecture/CRM_AUTH_SERVICE_SA.md)

## 2. 角色與權限

這個功能誰能用、誰不能用、需要哪個 permission code(參照既有 catalog,不要臨時發明命名規則)。

| 角色 | 可以做什麼 |
|---|---|
| | |

## 3. 操作流程(正常路徑)

用步驟或流程圖描述使用者怎麼一步步完成這件事。

## 4. 狀態轉換

如果這個功能牽涉到某個實體的狀態變化(例如訂單狀態、匯入批次狀態),畫出狀態轉換規則。

## 5. 例外流程

- 使用者沒有權限時會發生什麼?
- 資料不存在時會發生什麼?
- 操作衝突(例如兩人同時編輯)時會發生什麼?

## 6. 資料流與敏感資料

- 資料從哪裡來、經過哪些系統、存到哪裡?
- 是否有 Restricted 等級資料流動(參照 PRD 第 7 節)?
- 是否有資料離開受控環境的風險(例如匯出、傳給外部 AI)?

## 7. 與外部系統的互動

（如果有,例如呼叫 AuthService 驗證 token、呼叫外部 ERP API)

## 8. 風險點與人工審核點

哪些步驟風險高,需要人工確認而不是全自動?（例如客戶合併、權限異動)

## 9. 待決策事項

列出 SA 階段發現、需要 Steven 決定的問題。
```

## 使用方式

1. AI 根據已核准的 PRD 草擬 SA。
2. 優先引用既有架構文件(`docs/architecture/`),不重複描述已經寫過的系統邊界。
3. Steven 重點看第 2(權限)、5(例外)、8(風險點),這幾節如果有疑慮就是要卡關的地方。
4. 確認後進入 [03-sd-template.md](./03-sd-template.md)。
