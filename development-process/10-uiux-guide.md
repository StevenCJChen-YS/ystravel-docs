# UI/UX 設計流程指南

> 這份補上原本流程缺的一塊：**UI/UX 怎麼在流程裡落地**。你（Steven）是 UI/UX 主導者，這份文件講「新畫面從想法到實作」該經過什麼、什麼可以省。

## 1. 兩個來源，先分清楚

| 文件 | 是什麼 | 何時看 |
|---|---|---|
| `docs/design-system/foundation.md` | 全公司共用設計系統（顏色/字體/密度/按鈕…）＋ 各系統主色 | 做任何畫面前的地基，不常改 |
| 本文件 | 「單一功能」做畫面的流程步驟 | 每次新功能的 UI 階段 |

**原則：畫面一律用 design-system 的 token 與 NuxtUI 元件，不各頁自己發明顏色/尺寸。**

## 2. 一個新畫面該走的步驟（輕量版）

```
PRD 第 6 節「UI/UX 重點」列畫面清單
  -> （視情況）Wireframe 線框：只排版、不上色
  -> 直接用 NuxtUI + design-system token 做出來
  -> 你手動看畫面 QA、邊做邊調
  -> 對照 PRD 的 Acceptance Criteria 驗收
```

## 3. 哪些產物要做、哪些跳過（給你的情況量身）

| 產物 | 做不做 | 說明 |
|---|---|---|
| **設計 Token 規範** | ✅ 一次做好（見 design-system） | 全系統受惠，不每頁重來 |
| **Wireframe（低擬真線框）** | ✅ 建議做，用最快方式（手繪/拉方塊即可） | 你是視覺型，開工前先確認版面能省重工 |
| **Hi-Fi Mockup（精美設計稿）** | ❌ 跳過 | 你習慣直接在 NuxtUI 上邊做邊調，畫精美稿是浪費 |
| **Clickable Prototype** | ❌ 現階段不用 | MVP 不需要 |

> 例外：如果某個畫面互動特別複雜（多步驟、多狀態切換），值得先畫個簡單流程圖或線框對齊，再動手。

## 4. 每個畫面一定要想的「四種狀態」

做畫面時，別只做「有資料、正常」那一種。至少想過：

1. **Loading**（讀取中）
2. **空狀態**（沒有資料時顯示什麼、引導使用者做什麼）
3. **錯誤**（讀取失敗/沒權限，訊息怎麼呈現）
4. **正常**（有資料）

這幾個在 PRD 第 6 節就先列出來，避免上線後才發現「沒資料時畫面一片空白」。

## 4.5 訊息回饋：inline 還是 toast（全站通用）

- **前端就能判斷的錯**（欄位格式、必填未填）→ 顯示在**該 input 欄位下方**（即時、不打斷操作）。
- **需送後端才知道的結果**（成功/失敗、伺服器錯誤）→ 用 **toast**（成功 toast 後再導轉；失敗顯示可讀訊息）。
- 這是「呈現方式」，屬 UI/UX，**不寫進 Gherkin**。Gherkin 只描述**訊息內容/行為**（例如「顯示帳號或密碼錯誤」），不綁定用 toast 還是別的元件。

## 5. PRD 第 6 節「UI/UX 重點」怎麼填（給 AI 的提示）

AI 草擬 PRD 時，第 6 節至少列：
- 需要哪些畫面 / 狀態（含上面四種狀態）
- 特別要注意的互動或例外
- 可沿用的既有元件（參照 `design-system/foundation.md` §4、AuthPortal shared UI 清單）
- 這個功能屬於哪個系統 → 用該系統的主色（Auth=Blue / CRM=Violet / EIP=Teal）

## 6. 什麼時候可以完全跳過這份流程

- 純視覺微調（顏色、間距、文字）：直接改、直接看畫面，不需要任何 UI 文件（對齊 `00-overview.md` §5）。
- 這份流程只在「**新畫面 / 新功能的畫面**」時才需要走。

## 7. 驗收（誰簽核）

- 畫面實作完，由**你**手動操作確認（這是你最擅長、也最終負責的部分）。
- 對照 PRD 的 Acceptance Criteria 走一次；四種狀態都看過。
- 有 dark mode 的畫面，light / dark 都要看過。
