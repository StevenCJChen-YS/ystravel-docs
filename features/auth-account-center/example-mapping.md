## 導覽與唯讀性

### Rule 1｜個人資料頁整頁唯讀，所有編輯集中在設定頁
- ✅ 在 `/account/profile` 看到名片 hero＋公司資料，畫面上沒有任何可編輯控件
- ✅ 點「前往設定」跳到 `/account/settings`，編輯功能都在這

### Rule 2｜公司資料本人唯讀，由管理員維護
- ✅ 部門／職稱／角色／帳號欄位旁標示「由管理員維護」，本人無法編輯
- ❌（邊界）本人即使用開發者工具，也沒有可打的公司資料寫入 API（後端不開放本人寫入這些欄位）

## A. 大頭貼

### Rule A1｜只有本人能改自己的頭貼
- ✅ Amy 上傳並裁切照片 → Amy 自己的名片頭貼更新
- ❌ Amy 不能替 Bob 改頭貼（後端只認 token 內的本人 id，不吃外部指定的 targetUserId）
- ❌ 管理員也不能替員工改頭貼（Phase 1 明確不做，PRD §8）

### Rule A2｜只接受白名單圖片格式，且不超過大小上限
- ✅ 上傳 JPG／PNG／WebP 且在大小上限內 → 成功
- ❌ 上傳 PDF、或把副檔名改成 `.jpg` 的假圖 → 明確錯誤、不儲存（後端驗真實內容 magic bytes，不只看副檔名）
- ❌ 上傳超過大小上限的檔 → 明確錯誤；畫面事先顯示限制值

### Rule A3｜頭貼變更後跨處即時同步且持久
- ✅ 儲存後：頁面 hero、右上 UserMenu 頭貼立即換；重新整理後仍在；其他同事看到的名片也更新

### Rule A4｜可移除頭貼回到預設
- ✅ 已有頭貼者點「移除頭像」→ 回到預設灰頭像
- ✅ 從未上傳頭貼者維持預設，畫面不壞版

### Rule A5｜頭貼變更留 audit（不存圖片內容）
- ✅ 上傳成功寫一筆 audit（誰、何時、事件類型）
- ✅ 移除頭貼也寫一筆 audit（Q2 已定：移除同樣留紀錄）
- ✅ 任何 audit／log 內都不含圖片二進位內容

## C. 自助改密碼

> 前置事實（2026-07-15 查證 AuthService code 落檔）：現行帳號啟用**唯一路徑**＝管理員建帳號（`status: INVITED`、`passwordHash: null`、寄邀請信）→ 員工點邀請信連結設第一組密碼 → 才變 `ACTIVE`；Google 登入強制要求帳號已是 `ACTIVE`。因此**現況不存在「純 Google、從未設過密碼」的活躍帳號**——每個能用 Google 登入的人一定早就設過密碼。Rule C4 因此是防禦性設計（未來若開放 Google 直接啟用、或遺留資料 `passwordHash` 為 null 才會遇到）。

### Rule C1｜改密碼需驗證目前密碼，新密碼須過既有密碼政策
- ✅ 目前密碼正確＋新密碼合政策＋兩次一致 → 改密成功
- ❌ 新密碼 <12 碼／強度不足／含個資型／與現用相同 → 即時提示、擋下
- ❌ 兩次新密碼不一致 → 擋下

### Rule C2｜目前密碼錯誤要有頻率限制（防暴力試密）
- ❌ 目前密碼錯誤 → 明確提示「目前密碼不正確」
- ❌（邊界）短時間內連續多次錯誤 → 觸發頻率限制、暫時擋下並留 audit

### Rule C3｜改密成功撤銷其他裝置、保留本機
- ✅ 改密成功 → 其他裝置的 refresh token 全撤、本機仍登入；畫面提示「其他裝置已登出」
- 註（Q3 已定）：連「記住我」的長期 token 也一起撤，只留本機當前 session

### Rule C4｜沒有密碼的帳號顯示合理替代狀態（防禦性）
- ✅ 帳號安全區塊依「是否有 passwordHash」決定呈現：有 → 顯示改密表單；沒有 → 顯示替代狀態文字（例如「你目前使用 Google 登入」），不出現要填「目前密碼」卻無密碼可填的壞表單
- ✅ 替代狀態文字附一句指引：可用「忘記密碼」流程設定一組密碼（重用現有流程，Phase 1 不另做設定密碼入口）

### Rule C5｜密碼事件留 audit，絕不記錄明文
- ✅ 改密成功／失敗／觸發頻率限制各留 audit
- ✅ 任何 log／audit 都不含密碼明文
