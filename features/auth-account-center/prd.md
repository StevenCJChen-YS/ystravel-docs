## 1. 一句話摘要

把 AuthPortal 的個人世界拆成兩頁：「**個人資料**」（唯讀：身份名片＋公司資料，取代現在的佔位頁）與新的「**設定**」頁（所有編輯集中在這：換大頭貼含裁切、挑名片背景、自助改密碼）；公司資料（部門/職稱/角色）維持唯讀由管理員管理。

## 2. 背景與動機

### 2.1 現況問題

- `/account/profile` 目前是**唯讀佔位頁**：只顯示顯示名稱、帳號、角色，右欄放「規劃中」文案，沒有任何可操作功能。
- 身份名片視覺（背景圖＋頭像＋名字＋角色）內嵌在 `UserMenu.vue` 的 popover 裡，頭像固定用預設圖、背景固定一張日本風景——**使用者完全沒有個人化空間**。
- 已登入的使用者**沒有自助改密碼的管道**：忘記密碼流程（auth-password-reset）解決的是「被鎖在門外」，但「我想主動換密碼」目前只能假裝忘記密碼繞路。

### 2.2 資訊架構：「看／改」分頁 ＋ 「兩桶模型」分區塊（2026-07-15 拍板）

**第一軸：看的頁 vs 改的頁**（GitHub / Microsoft myaccount 式）：

| 頁 | route | 性質 |
|---|---|---|
| **個人資料** | `/account/profile` | **整頁唯讀**：身份名片 hero＋公司資料展示；頁上有「前往設定」入口 |
| **設定** | `/account/settings` | **所有編輯集中在這**：換頭貼、換背景、改密碼，未來的綁定/偏好也進這頁 |

選這個切法的理由：①與 SSO 唯讀特性（§2.3）天生合拍——個人資料頁大半本來就不可改，整頁唯讀名正言順，不會「這欄能改那欄不能改」混亂 ②設定頁會隨功能自然長大（LINE、顯示設定…），架構不用再動 ③每個編輯功能只有一個入口，好維護。

**第二軸：兩桶模型（2026-07-12 拍板）**——設定頁**內部**的區塊分類，未來所有個人功能先問「屬於哪一桶」：

| 桶 | 內容 | 例子 |
|---|---|---|
| 🪪 個人化（identity） | 「我看起來如何」 | 大頭貼、名片背景、（未來）顯示名稱 |
| 🔐 帳號安全（security） | 「我的帳號安不安全」 | 改密碼、（未來）外部帳號綁定 Google/LINE、2FA、登入裝置 |

**升級規則**：設定頁現在內容少＝單頁分區塊；等任一桶長胖到 2~3 個區塊，再升級左側 tabs（或拆 `/account/security`，route 已預留）。不要一開始就做空蕩蕩的多頁架構。

### 2.3 SSO 唯讀特性（本 PRD 正式落檔）

AuthPortal 是 SSO 身分中樞，個人資料頁**大部分欄位是公司資料、本人唯讀**：

- 部門、職稱、角色、帳號（username/email）由**管理員／未來 HR 主檔**維護，本人只能看、不能改；畫面要明示「由管理員維護」。
- 本人能自己改的只有：**大頭貼＋名片背景**（本 Phase）；顯示名稱是否開放自改**未定案**（見 §8）——若未來開放，稽核紀錄與官方列表仍一律用本名＋帳號，避免頂著暱稱做管理操作。

### 2.4 為什麼現在做

- 大廳（系統總覽）dark 改版剛完成，個人世界（LobbyShell 這一掛）只剩個人資料頁還是佔位頁，體驗斷層明顯。
- 身份名片是**跨系統共用元件**（未來 CRM/EIP 都要顯示同一張名片，`AUTHPORTAL_UI_FOUNDATION.md` §10），早點把資料模型（avatarUrl/backgroundKey）定下來，之後其他系統直接吃。

### 2.5 通用偏好的架構方向（本 PRD 落檔方向、不實作）

使用者顯示偏好分兩種，**存的地方不同**（2026-07-15 拍板）：

| 偏好類型 | 例子 | 存哪裡 |
|---|---|---|
| **通用偏好**（跟「人」走，到哪個系統都一樣） | 主題明暗、**介面縮放**（EIP 經驗：一半以上使用者年紀較大需要放大） | **Auth 帳號層級**儲存，各系統讀取後自行套用 |
| **系統專屬偏好**（只對該系統有意義） | CRM 表格密度、EIP 通知方式 | 各系統自己的設定頁 |

原則：SSO 是「使用者的家」，「這個人喜歡怎麼看東西」存帳號上，設一次到處生效、換裝置也跟著人。技術註：全站 Tailwind rem 單位，調 root 字級即可整體縮放。**Phase 1 不實作**——帳號層級偏好需要新的偏好儲存 API＋各系統接入，獨立成 feature（暫名 `auth-user-preferences`）排後續；主題切換維持現況（UserMenu、裝置層級）。

## 3. 目標使用者

- **全體內部員工**：每人都有的個人頁；換頭貼/背景屬低頻但情感價值高（「這是我的帳號」）；改密碼低頻但必要。
- **管理員**：不新增管理功能，但受益於「員工可自助改密碼」＝又少一件代辦；稽核頁可看到頭像/密碼變更事件。
- 使用頻率：初次設定一波，之後低頻。

## 4. User Story

**A（大頭貼）**
作為內部員工，
我希望在個人資料頁上傳自己的照片並裁切成頭像，
以便在所有系統（Auth/CRM/EIP）的名片與選單上都顯示我的臉，而不是預設灰頭像。

**B（名片背景）**
作為內部員工，
我希望從公司提供的風景圖中挑一張當我的名片背景，
以便讓個人名片有一點個人風格，同時整體視覺仍維持公司品牌調性。

**C（自助改密碼）**
作為內部員工，
我希望在帳號安全區塊輸入目前密碼與新密碼後直接完成變更，
以便主動維護帳號安全，不用走「忘記密碼」繞路或找管理員。

## 5. Acceptance Criteria（驗收標準）

**個人資料頁（`/account/profile`，唯讀）**
- [ ] 頁面標題由「個人資料管理」改為「**個人資料**」；整頁唯讀、無任何編輯控件
- [ ] 頂部是**身份名片 hero**：名片背景圖＋大頭貼＋顯示名稱＋角色/帳號，與 UserMenu popover 的名片視覺一致（同一個共用元件）
- [ ] 名片下方有「公司資料」唯讀區塊：帳號、角色（未來加部門/職稱），欄位旁標示「由管理員維護」
- [ ] 頁上有明顯的「前往設定」入口（去 `/account/settings`）
- [ ] 頁面留在 LobbyShell（大廳輕頂欄），dark 模式延續大廳視覺語言、light 維持白卡規範

**設定頁（`/account/settings`，新頁）**
- [ ] 新路由 `/account/settings`（僅需登入），頁面標題「設定」；UserMenu 選單新增「設定」項（順序：個人資料／設定／切換主題／登出）
- [ ] 單頁分區塊：「🪪 個人化」（更換大頭貼、更換名片背景）＋「🔐 帳號安全」（修改密碼）
- [ ] 在設定頁改了頭貼/背景後：個人資料頁 hero、右上 UserMenu 名片同步更新（同一份資料）
- [ ] 同樣留在 LobbyShell，深淺模式規範同上

**A 大頭貼上傳（在設定頁）**
- [ ] 個人化區塊有「更換頭像」入口；選擇圖片後出現**正方形裁切框**（可縮放、拖曳），確認後儲存
- [ ] 只接受常見圖片格式（JPG/PNG/WebP），檔案過大或格式不符時有明確錯誤訊息（上限值由 SD 定，畫面要顯示限制）
- [ ] 儲存成功後：頁面 hero、右上 UserMenu 頭像立即更新，重新整理後仍在；其他使用者看到的名片也更新
- [ ] 未上傳過頭像的人維持現在的預設頭像，不會壞版
- [ ] 可以「移除頭像」回到預設圖
- [ ] 頭像變更留 audit log（誰、何時；不存圖片內容進 log）

**B 名片背景選擇（在設定頁）**
- [ ] 個人化區塊有「更換名片背景」入口，開啟後以縮圖清單顯示**公司提供的 4~6 張旅遊風景**（既有日本背景為第一張、也是預設值）
- [ ] 點選縮圖即可預覽、確認後儲存；儲存的是背景代號（backgroundKey），**不開放自訂上傳背景**
- [ ] 儲存後個人資料頁名片 hero 與 UserMenu popover 背景同步更新，重新整理後仍在
- [ ] 沒選過的人顯示預設背景（日本），不會壞版

**C 自助改密碼（在設定頁）**
- [ ] 設定頁「帳號安全」區塊內含「修改密碼」入口
- [ ] 修改密碼需輸入：**目前密碼＋新密碼＋確認新密碼**；新密碼須通過**既有密碼政策**（單一真相來源＝`password-policy.ts`：目前 `MIN_PASSWORD_LENGTH=10`、zxcvbn 強度檢查、擋個資型、不可與現用相同），畫面有即時提示（沿用登入/重設頁的密碼提示體驗）〔2026-07-15 修正：原寫「≥12」，改密沿用既有政策值以保持全站一致，見 sd.md §10 D2〕
- [ ] 目前密碼錯誤時明確提示，且有頻率限制（防止拿這個表單暴力試密碼）
- [ ] 改密成功後：**其他裝置全部登出（撤銷其他 refresh token）、本機保留登入**；畫面提示「其他裝置已登出」
- [ ] 純 Google 登入、尚未設定過密碼的帳號：此區塊顯示合理狀態（例如「你目前使用 Google 登入」），不出現會報錯的表單（細節 SD 議）
- [ ] 密碼變更成功/失敗（含頻率限制觸發）留 audit log，**絕不記錄密碼明文**

## 6. UI / UX 重點

（由 Steven 主導，先列畫面/狀態清單，細節做的時候看畫面調）

- **身份名片 hero**：從 `UserMenu.vue` popover 抽出成共用元件 `UserIdentityCard`（`AUTHPORTAL_UI_FOUNDATION.md` §13 既定命名），支援 popover 小尺寸與頁面 hero 大尺寸兩種呈現；背景圖上維持左→右黑色漸層薄紗保文字可讀
- **個人資料頁（唯讀）**：名片 hero → 公司資料區塊（欄位標「由管理員維護」）→「前往設定」入口（位置/樣式 Steven 看畫面定）
- **設定頁**：單頁區塊順序＝🪪 個人化（更換頭像、更換名片背景）→ 🔐 帳號安全（修改密碼）；每個設定項用「標題＋說明＋動作鈕」的列式排版（業界慣例，好掃視、未來加項目直接往下疊）
- **更換頭像**：modal 流程（選檔 → 裁切 → 預覽 → 儲存），走既有 FormModal 範式；loading／上傳失敗／檔案太大三種狀態要有
- **更換背景**：輕量選擇器（popover 或 modal 縮圖 grid），選中態明顯
- **改密碼**：modal（三欄位＋密碼強度即時提示），成功 toast＋「其他裝置已登出」說明
- **空/邊界狀態**：無頭像＝預設圖；無背景＝預設日本；Google-only 帳號的密碼區塊替代文案
- 既有元件沿用：FormModal、AppInput、UserMenu、LobbyShell、AppPageLayout；控件尺寸照 `docs/reference/AUTHPORTAL_NUXTUI_COMPONENT_SCALE.md`

## 7. 資料分級與風險（必填 — 依 SECURITY_AND_ISO27001_BASELINE.md §6.1）

| 欄位 | 內容 |
|---|---|
| 涉及的資料類型 | 員工大頭貼影像（可識別個人）、名片背景代號、密碼（僅雜湊，流程中出現明文輸入） |
| 資料分級 | 大頭貼＝Internal（員工內部資料，非旅客個資，但屬可識別影像，僅登入使用者可見）；backgroundKey＝Internal；密碼＝最高等級機密（等同 Restricted 處理：不落 log、不回傳、僅存雜湊） |
| 誰可以看 | 大頭貼/背景：所有已登入內部使用者（名片本來就是給同事看的）；密碼：任何人（含管理員）都不可見 |
| 誰可以匯出 | 不提供匯出功能 |
| 誰可以編輯 | 僅本人可改自己的大頭貼/背景/密碼；管理員**不能**替人改頭貼/背景（如未來有不當圖片需求，另議管理員「移除頭像」權限——列 §9） |
| 是否涉及 AI | 否 |
| 若涉及 AI，AI 會接觸哪些欄位 | — |
| 風險摘要 | ①上傳檔案是新攻擊面：偽裝圖片的惡意檔、超大檔打爆儲存——SD 必須定型別白名單、大小上限、重新編碼（re-encode）策略 ②不當頭像（冒犯性圖片）目前無審核機制，公司內部先靠行政管理，必要時補管理員移除功能 ③改密碼表單可能被拿來暴力試「目前密碼」——必須有頻率限制與 audit ④密碼明文絕不落 log/audit |

## 8. 不做的事（Out of Scope）

- **自改顯示名稱**：未定案（2026-07-15 Steven 未列入）。若未來開放，鐵則＝稽核與官方列表仍用本名＋帳號。
- **LINE 綁定／通知綁定**：完全沒有規劃文件，另開 feature。**設定頁的「連結帳號」區塊（含 Google 綁定狀態顯示）等 LINE 那輪一起做**（2026-07-15 拍板：避免 Phase 1 留空殼區塊、Google 狀態顯示也要新後端端點）。
- **顯示設定／介面縮放（通用偏好）**：架構方向已落檔（§2.5：Auth 存帳號層級通用偏好、各系統套用），實作另開 feature `auth-user-preferences`；主題切換維持 UserMenu 現況。
- **通知設定頁（/account/notifications）**：未來再議。
- **2FA / TOTP**：維持 P5 暫緩（`auth-native-login-v2` 既有決策）。
- **Google 自助綁定/解綁 UI**：目前 Google 綁定由首登自動建立，自助管理另議。
- **UserIdentityCard 鋪到 CRM/EIP**：本輪只在 AuthPortal 落地；跨系統顯示等 CRM 端有畫面再接（資料模型這輪就定好）。
- **背景自訂上傳**：明確不做（§10 既定決策：固定素材庫）。
- **登入裝置清單／單裝置登出管理**：security 桶未來項目。
