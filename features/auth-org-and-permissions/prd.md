## 1. 一句話摘要

把 AuthService 的使用者資料模型補完：結構化的公司→部門組織、群組（名單）、暫時性角色、SUPER_ADMIN 機制與使用者欄位增改，並提供對應的後台管理介面，取代 EIP 的字串部門、tag 權限 hack 與雙表員工設計。

## 2. 背景與動機

- EIP 的 user/employee 雙表鬆散關聯、部門用純文字、用「tag 權限」反查處理人員名單（超管因擁有所有權限被誤列入所有名單）——這些設計債已實際造成維運混亂。
- CRM 之後要做「只看自己部門的客戶」這類**資料範圍權限**，前提是組織歸屬是結構化事實；現在 AuthUser.department 只是字串，做不到。
- 未來 HR 模組、LINE 綁定、統一通知層都依賴本輪定下的地基（單一人表、Group 名單、AuthIdentity）。愈晚做，之後要遷移的資料愈多。

## 3. 目標使用者

- **系統管理員（Steven）**：管理組織結構、群組、角色指派；日常使用。
- **人資（HR 角色）**：建立新員工帳號並寄出邀請信（第一階段的入職入口）。
- **全體員工**：間接受益——部門資訊正確、名單不再誤列。

## 4. User Story

1. 作為**系統管理員**，我希望在後台維護公司與部門結構，以便所有系統共用同一份組織事實。
2. 作為**人資**，我希望自己就能建立新員工帳號（選部門、填 email）並自動寄出邀請信，以便入職流程不再卡在系統管理員身上。
3. 作為**系統管理員**，我希望建立「群組」並明確管理成員名單，以便通知對象與處理人員名單不再從權限反推、超管不再被誤列。
4. 作為**系統管理員**，我希望指派角色時可以設定有效期，以便代理主管這類暫時性授權到期自動失效。
5. 作為**系統管理員**，我希望有一個不會被誤設定的超級管理員機制，以便系統永遠有一把行為明確的最後鑰匙。

## 5. Acceptance Criteria（驗收標準）

組織結構：
- [ ] 後台可新增/編輯/停用公司與部門（部門掛在公司下，可排序）
- [ ] 建立/編輯使用者時，部門改為下拉選單（一般員工必選）；email 為必填
- [ ] 既有使用者資料遷移後，原字串部門正確對應到新部門資料，畫面顯示無缺漏
- [ ] 使用者列表可依公司/部門篩選

群組：
- [ ] 後台可建立群組、加入/移除成員（成員一個一個明確加入）
- [ ] 群組成員名單與權限完全無關——擁有全部權限的帳號**不會**出現在任何未加入的群組名單中

角色：
- [ ] 指派角色可設定有效期；到期後該角色的權限立即不再生效（可實測驗證）
- [ ] 角色指派紀錄顯示指派人與時間

SUPER_ADMIN 與 HR 角色：
- [ ] 持有 SUPER_ADMIN 角色的帳號可通過所有權限檢查（即使未逐一持有個別權限）
- [ ] SUPER_ADMIN 的指派/移除有 audit 紀錄
- [ ] HR 角色帳號登入後：能建立使用者並寄邀請信；不能進入組織/角色/群組等其他系統管理功能

使用者欄位：
- [ ] 使用者可設定備註（note）；avatar 欄位就緒（上傳介面不在本輪）

## 6. UI / UX 重點

- 需要的畫面 / 狀態：
  - **組織管理頁**（新）：公司清單 + 展開部門，新增/編輯/停用/排序
  - **群組管理頁**（新）：群組清單 + 成員管理（加人用使用者搜尋選擇器）
  - **使用者管理頁**（改）：建立/編輯表單換部門下拉、加備註欄；列表加公司/部門欄與篩選；角色指派對話框加有效期欄位
- 特別要注意的互動或例外畫面：停用有成員的部門/群組時的警示；角色到期的顯示（列表標示「已過期」）；遷移後部門為空的使用者要醒目提示補填
- 既有元件沿用：AdminUsersPage 的表格/抽屜模式、badge、AUTHPORTAL_UI_FOUNDATION 規範；新頁面按 design-system foundation.md（含 §4.1 表單規則）
- 依 UI 待美化清單規則：AI 做完第一版 UI 後主動加入清單待 Steven 視覺巡檢

## 7. 資料分級與風險（依 SECURITY_AND_ISO27001_BASELINE.md §6.1）

| 欄位 | 內容 |
|---|---|
| 涉及的資料類型 | 員工姓名、公司 Email、部門/公司歸屬、職稱、管理備註、角色與權限指派 |
| 資料分級 | Internal（員工目錄資料）；其中 Email 依 baseline §3 預設 Restricted——本功能無匯出，僅後台顯示與管理 |
| 誰可以看 | 具 Auth 後台對應權限者（使用者管理/組織管理/群組管理各自權限） |
| 誰可以匯出 | 無匯出功能（本輪不做） |
| 誰可以編輯 | 使用者：USER.MANAGE（含 HR 角色）；組織/群組/角色：管理權限持有者（權限代碼粒度見 SA） |
| 是否涉及 AI | 否 |
| 風險摘要 | ①權限模型是全系統授權判斷的心臟，改壞＝所有 app 的授權失效，必須有完整測試與遷移演練 ②遷移失誤＝使用者部門歸屬遺失 ③SUPER_ADMIN 誤指派＝全權後門——以 audit 紀錄與指派限制防護 |

## 8. 不做的事（Out of Scope）

- **HR 模組本體**（人事資料、到職日等——未來另立系統，單向引用 userId）
- **CRM 資料範圍權限的實際判斷**（本輪只提供組織事實；CRM 側的「只看自己部門客戶」是 CRM 未來的功能輪）
- **LINE 綁定 / AuthIdentity 擴充**（等 LINE Webhook 輪）
- **統一通知層**（Group 表為它鋪路，但通知系統本身不在本輪）
- **avatar 上傳與顯示介面**（欄位進 schema，介面延後）
- **職稱管理**（title 維持字串）、**部門多層樹狀**、**群組發權限**（群組純名單）

## 9. 待釐清問題

- [ ] 後台管理權限的代碼粒度（組織/群組/角色管理是一顆 `AUTH.ADMIN.MANAGE` 還是拆開）——SA 給建議
- [ ] HR 角色的確切權限集（只有建使用者＋寄邀請？能不能停用使用者？）
- [ ] username 建帳號慣例（建議填員編）——實作時在建立表單預設帶入
- [ ] 既有使用者的 email 空值盤點結果（遷移前檢查）
