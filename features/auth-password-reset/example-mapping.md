# Example Mapping: auth-password-reset

> PRD 核准後、SA 之前的規則對齊。Rule = 業務規則；Example = 具體情境（含反例與邊界）；Question = 需 Steven 拍板的問題。
> 狀態：已確認（Steven 2026-07-09）

## Rule 1：忘記密碼申請不得洩漏帳號存在性

- Example：啟用中帳號 `amy@ystravel.com.tw` 提交忘記密碼 → 畫面顯示「若此信箱有註冊帳號，重設連結已寄出」，且 Amy 收到重設信
- Example：不存在的 `nobody@ystravel.com.tw` 提交 → 畫面顯示**同一句話**，不寄任何信
- Example：停用（INACTIVE）帳號提交 → 畫面顯示同一句話，不寄信，留 audit
- Example（邊界）：輸入 `  AMY@Ystravel.com.tw ` （大小寫混雜＋前後空白）→ 正規化後視同 `amy@ystravel.com.tw` 正常處理

## Rule 2：寄信有頻率限制，防信箱轟炸

- Example：同一 email 60 秒內提交第二次 → 畫面仍顯示成功句，但不再寄信
- Example：同一 email 一小時內第 6 次申請 → 不寄信，留 audit（`RESET_RATE_LIMITED`）
- 預設值：同一 email 60 秒 1 封、每小時最多 5 封（env 可調）

## Rule 3：重設連結 30 分鐘內單次有效，且一人同時只有一個有效連結

- Example：Amy 收信後 10 分鐘內點連結 → 進入重設頁可設新密碼
- Example：31 分鐘後才點 → 顯示「連結已失效」＋「重新申請」入口
- Example：重設成功後再點同一連結 → 顯示「連結已失效」
- Example（邊界）：Amy 連續申請兩次收到兩封信 → 只有**最新一封**的連結有效，舊連結點開顯示失效

## Rule 4：新密碼必須通過 P3 密碼政策

- Example：設定 `a12345678`（9 碼、弱）→ 拒絕，顯示具體原因（沿用 P3 訊息與前端即時提示）
- Example：設定與**現用密碼相同**的新密碼 → 拒絕「新密碼不可與目前密碼相同」
- Example：設定 `X7#mQ2vLp9$wK4`（合規）→ 成功

## Rule 5：重設成功後該帳號全裝置登出

- Example：Amy 在公司電腦與家用電腦都登入中，在家重設密碼成功 → 兩台的 session 全部失效（refresh token 全撤銷），需用新密碼重新登入
- Example：重設成功後用**舊密碼**登入 → 失敗（統一模糊訊息）；用**新密碼**登入 → 成功

## Rule 6：新帳號走邀請流程，設定密碼前不可登入

- Example：管理員建立 `newbie@ystravel.com.tw`（無密碼欄位）→ 系統寄邀請信，列表顯示「邀請中」
- Example：Newbie 72 小時內點連結設定合規密碼 → 完成，可用該密碼登入，列表轉「啟用」
- Example：73 小時後才點 → 顯示「邀請已過期，請聯絡管理員」
- Example：管理員按「重發邀請」→ 新信寄出、舊連結立即失效、72 小時重新起算
- Example（邊界）：邀請中帳號直接嘗試帳密登入 → 失敗，統一模糊訊息（不提示「尚未設定密碼」）

## Rule 7：管理員只能「觸發寄信」，不能接觸密碼

- Example：具使用者管理權限的管理員對啟用中帳號按「發送重設連結」→ 該員工收到 30 分鐘重設信，留 audit（含操作者）
- Example：後台建帳號與重設兩處**皆無密碼輸入欄位**
- Example：無使用者管理權限的人呼叫發送重設連結 API → 403 拒絕
- Example（邊界）：對停用帳號按「發送重設連結」→ 拒絕（按鈕隱藏/禁用，API 也擋）

## Rule 8：關鍵事件必留 audit，且絕不記錄密碼與 token 明文

- Example：寄出重設信/邀請信、重設成功、連結失效或重放、頻率限制觸發、管理員觸發寄信 → 各留一筆 audit（含 email、IP、操作者）
- Example：任何 audit 內容中不出現密碼明文或連結 token

## Rule 9：邀請中帳號提交忘記密碼 → 重寄邀請信

- Example：邀請中帳號 `newbie@ystravel.com.tw` 在忘記密碼頁提交 → 畫面顯示統一成功句，收到的是**邀請信**（72 小時效期），舊邀請連結失效
- Example：頻率限制與 Rule 2 共用同一組計數
