## 1. 一句話摘要

讓員工可以在登入頁自助「忘記密碼」透過 email 重設密碼，同時把後台管理員流程改成「永遠不碰明文密碼」：建帳號改寄設定密碼邀請信、幫人重設改成發送重設連結。

## 2. 背景與動機

- P3 密碼政策（12 碼 + 強度檢查）已上線，但**忘記密碼目前完全沒有自助管道**——員工忘記密碼只能找管理員，管理員再幫他「設一組新密碼然後用嘴巴/訊息告訴他」。
- 這個現況有兩個問題：
  1. **管理員知道使用者的明文密碼**。資安原則是除了本人，任何人（包括 IT）都不該知道你的密碼；密碼用訊息軟體傳遞更是外洩風險。`PASSWORD_POLICY.md` §5 早已把「管理員不得直接得知或設定使用者明文密碼」列為目標態。
  2. 員工忘記密碼要等管理員有空，體驗差、也增加管理員雜務。
- AuthService 目前**沒有任何寄信能力**，這輪會一併建立寄信基礎建設（之後的通知類功能都能重用）。

## 3. 目標使用者

- **內部員工**：忘記密碼時自助重設；新進員工收邀請信自己設定第一組密碼。
- **IT / 管理員**：建帳號、幫忘記密碼又收不到信的員工重發連結；全程不再接觸任何人的明文密碼。
- 使用頻率：低頻但關鍵（忘記密碼是「被鎖在門外」情境，做不好會直接打電話找 IT）。

## 4. User Story

**A（自助重設）**
作為內部員工，
我希望在登入頁點「忘記密碼」、輸入公司 email 後收到重設連結，自己設定新密碼，
以便不用找管理員就能重新進入系統。

**B（管理員發送重設連結）**
作為管理員，
我希望在後台按一顆「發送重設連結」按鈕就好，
以便協助員工重設密碼時，我從頭到尾不知道也不經手他的密碼。

**C（新帳號邀請）**
作為管理員，
我希望建立新帳號時不用替對方想密碼，系統自動寄「設定密碼」邀請信，
以便新員工到職時自己設定只有他知道的密碼。

## 5. Acceptance Criteria（驗收標準）

**自助忘記密碼**
- [ ] AuthPortal 登入頁有「忘記密碼」入口，點進去輸入 email 送出後，**無論該 email 存不存在**都顯示同一句話（例：「若此信箱有註冊帳號，重設連結已寄出」），不洩漏帳號存在性
- [ ] 存在且啟用的帳號會收到重設信；信中連結開啟後可設定新密碼（新密碼須通過 P3 密碼政策，畫面上有即時提示）
- [ ] 重設連結 **30 分鐘內有效、單次有效**：用過再點顯示「連結已失效」，過期同樣失效
- [ ] 重設成功後，該帳號**所有裝置全部登出**（所有 refresh token 撤銷），畫面導回登入頁，用新密碼可登入、舊密碼不可
- [ ] 停用（INACTIVE）帳號提交忘記密碼：畫面同樣顯示成功句，但**不寄信**
- [ ] 短時間重複提交同一 email 有頻率限制，不會被拿來轟炸員工信箱

**管理員去密碼化**
- [ ] 後台建立新帳號時**沒有密碼欄位**；建立成功後系統自動寄「設定密碼」邀請信給該員工
- [ ] 員工點邀請連結設定密碼後帳號才能登入；邀請連結效期較長（見 §9），過期後管理員可在後台重發
- [ ] 後台原「重設密碼」（管理員輸入新密碼）改為「**發送重設密碼連結**」按鈕，按下後寄信給該員工
- [ ] 全後台找不到任何可以輸入/看見別人密碼的地方
- [ ] 未完成密碼設定的帳號在列表上看得出狀態（例如「邀請中」），管理員能重發邀請

**通用**
- [ ] 寄信、重設成功、連結失效/重放、頻率限制觸發等關鍵事件都留 audit log（**絕不記錄密碼與 token 明文**）
- [ ] 本機開發用 Mailpit（Docker 假信箱）能看到完整信件內容，實測全流程

## 6. UI / UX 重點

（由 Steven 主導，以下先列需要的畫面/狀態，細節做的時候看畫面調）

- **AuthPortal 登入頁**：加「忘記密碼？」文字連結（位置、樣式 Steven 定）
- **忘記密碼頁**（新）：輸入 email → 送出 → 統一成功畫面（含「沒收到？檢查垃圾信 / 聯絡管理員」提示）
- **重設密碼頁**（新）：新密碼 + 確認密碼、P3 政策即時提示（沿用 AdminUsersPage 既有的密碼提示體驗）、成功畫面 → 導登入頁；連結失效時的專屬錯誤畫面（含「重新申請」入口）
- **設定密碼頁（邀請）**：與重設密碼頁共用版型，文案不同（「歡迎加入，請設定您的密碼」）
- **AdminUsersPage**：移除兩處密碼輸入；建帳號成功 toast 改「邀請信已寄出」；使用者列表加「邀請中」狀態與「重發邀請」；「重設密碼」按鈕改「發送重設連結」
- **信件模板 x2**：重設密碼信、邀請信（繁中、含按鈕連結與純文字連結、效期說明）

## 7. 資料分級與風險（必填 — 依 SECURITY_AND_ISO27001_BASELINE.md §6.1）

| 欄位 | 內容 |
|---|---|
| 涉及的資料類型 | 員工 email、密碼（僅以 bcrypt hash 儲存）、重設/邀請 token（僅以 hash 儲存）、audit log |
| 資料分級 | Restricted（密碼與 token 相關）/ Internal（email） |
| 誰可以看 | token 明文只出現在信件連結中，資料庫只存 hash；密碼任何人（含管理員、DB 管理者）都看不到明文 |
| 誰可以匯出 | 不提供匯出 |
| 誰可以編輯 | 使用者只能改自己的密碼；管理員只能「觸發寄信」，不能設定內容 |
| 是否涉及 AI | 否 |
| 風險摘要 | 做錯最壞情況：重設連結被猜出/攔截 → 帳號接管。緩解：token 高熵隨機、DB 只存 hash、30 分鐘單次有效、重設後全裝置登出、全程 HTTPS、寄信頻率限制、帳號枚舉防護 |

## 8. 不做的事（Out of Scope）

- 簡訊 / 2FA 驗證重設（P5 之後）
- 密碼歷史多筆比對（目前僅維持 P3 的「不可與現用相同」）
- 「首登強制改密」機制——**本輪直接消滅臨時密碼場景**（管理員無法設密碼），所以不需要此配套；若未來出現例外場景再議
- Email 樣板後台管理（樣板寫死在程式碼）
- 對外客戶（非員工）的註冊/重設
- 正式環境 SMTP 的實際串接（本輪只做到「換 env 設定即可接上」，公司 SMTP 資訊之後補）

## 9. 待釐清問題

- [x] 邀請信（設定密碼連結）效期：**72 小時**（Steven 2026-07-09 核准），過期後管理員一鍵重發。
- [ ] 既有「邀請中」概念與現有 `status`（ACTIVE/INACTIVE）如何呈現，SA 階段給方案。
