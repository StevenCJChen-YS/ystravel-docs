## 0. 白話摘要

這份在分析「員工自助重設密碼 + 新帳號邀請信 + 管理員去密碼化」會動到哪些系統、狀態怎麼流轉、哪裡有風險。關鍵結論：只動 AuthService（後端）與 AuthPortal（前端），CRM 完全不用改；帳號會多一個「邀請中」狀態；最大的風險是「重設連結本身就是一把臨時鑰匙」，所以連結對應的 token 在資料庫只存 hash、短效、單次有效，寄信入口有頻率限制。另一個新東西是 AuthService 第一次具備寄信能力（SMTP 抽象層），本機用 Mailpit 假信箱開發。

## 1. 系統邊界

- **涉及**：
  - `Ystravel-AuthService`：新增寄信模組、token 模型、忘記密碼/重設/邀請 API；修改建帳號與重設流程
  - `Ystravel-AuthPortal`：登入頁入口、忘記密碼頁、重設/設定密碼頁、AdminUsersPage 改造
  - `Ystravel-LocalDocker`：新增 Mailpit 容器（本機假信箱，非 git repo）
- **不涉及**：CRM 前後端、EIP、Google 登入流程（`auth-google-login`）皆不動
- 既有架構文件：`docs/features/auth-foundation/architecture-decisions.md` §0.6（native auth v2 主線）、`docs/features/auth-native-login-v2/sd.md`（RS256、refresh token、audit 既有設計）

## 2. 角色與權限

| 角色 | 可以做什麼 |
|---|---|
| 任何人（未登入） | 在忘記密碼頁提交 email（有頻率限制）；用有效連結重設/設定密碼 |
| 具 `AUTH.USER.MANAGE` 的管理員 | 建帳號（觸發邀請信）、發送重設連結、重發邀請 |
| 所有登入使用者 | 不變；本功能不新增登入後的自助頁面（改密碼走忘記密碼流程即可） |

不新增 permission code，沿用既有 `AUTH.USER.MANAGE`。

## 3. 操作流程（正常路徑）

**A. 自助忘記密碼**
1. 員工在 AuthPortal 登入頁點「忘記密碼？」
2. 輸入公司 email → 送出 → 畫面顯示統一成功句
3. 後端：正規化 email → 查帳號（啟用中才寄）→ 產生 token（舊的失效）→ 寄信
4. 員工收信點連結 → 前端先向後端驗證 token 有效 → 顯示重設頁
5. 設定新密碼（過 P3 政策）→ 後端寫入新 hash、撤銷全部 refresh token、token 標記已用
6. 導回登入頁，用新密碼登入

**B. 管理員發送重設連結**
1. AdminUsersPage 對啟用中帳號按「發送重設連結」→ 後端走與 A-3 相同的寄信路徑（audit 記操作者）

**C. 新帳號邀請**
1. 管理員建帳號（無密碼欄位）→ 帳號以「邀請中」狀態建立 → 自動寄邀請信（72 小時 token）
2. 員工點連結 → 設定密碼頁（邀請文案）→ 設定成功 → 帳號轉「啟用」
3. 過期未完成 → 管理員在列表按「重發邀請」

## 4. 狀態轉換

```text
（管理員建立）
   INVITED（邀請中，passwordHash 為空，不可登入）
      │ 員工完成密碼設定
      ▼
   ACTIVE（啟用，可登入）◄──────┐
      │ 管理員停用              │ 管理員重新啟用
      ▼                        │
   DISABLED（停用，不可登入）───┘
```

- 既有帳號（都有密碼）不受影響，維持 ACTIVE/DISABLED。
- INVITED 帳號：不可登入、不可被發「重設連結」（只能重發邀請）、可被停用（停用後邀請連結也失效）。
- DISABLED 帳號：不寄任何信。

## 5. 例外流程

- **email 不存在 / 帳號停用**：畫面一律統一成功句，不寄信（防帳號枚舉）
- **連結過期 / 已用 / 被新連結取代**：重設頁顯示「連結已失效」＋重新申請入口；不透露具體原因
- **新密碼不合政策**：顯示具體原因（此時使用者已證明持有有效連結，不算洩漏）
- **頻率限制觸發**：畫面仍顯示成功句，靜默不寄，留 audit
- **無權限呼叫管理員 API**：403
- **併發**：同帳號同時兩個有效連結不可能存在（發新即廢舊）；同一連結被兩個請求同時使用時，只有一個成功（token 單次使用以 DB 條件更新保證）
- **管理員自鎖風險**：super admin 忘記密碼也走同一自助流程（他的 email 收得到信即可自救）；若連信箱都進不去，仍可由另一位具 `AUTH.USER.MANAGE` 的管理員發送重設連結

## 6. 資料流與敏感資料

- Token 明文只存在於**信件連結**中，資料庫僅存 hash（同 P1 refresh token 的 selector/verifier 模式）；信件內容經 SMTP 傳輸
- 密碼明文只存在於使用者提交的 HTTPS request body，即刻 bcrypt hash 後落庫
- Restricted 資料流動：重設連結（等同臨時憑證）經過郵件系統——這是本功能固有風險，以短效、單次、hash 儲存、全裝置登出緩解
- 開發環境信件只進 Mailpit（本機容器），不會真的寄出

## 7. 與外部系統的互動

- **SMTP 伺服器**：本機 = Mailpit 容器（SMTP :1025 / Web UI :8025）；正式 = 公司 SMTP（資訊待補，只需換 env）
- 無其他外部呼叫

## 8. 風險點與人工審核點（Steven 重點看這節）

| 風險 | 說明 | 緩解 |
|---|---|---|
| 信件送達性 | 正式環境若被當垃圾信，員工收不到 → 打回 IT 原點 | 正式 SMTP 用公司網域寄件者；上線前實測收信；後台保留「發送重設連結」當人工後備 |
| 信箱被入侵 = 帳號被接管 | email 是重設的信任根，公司信箱被盜就能重設密碼 | 業界共同限制；30 分/單次/全裝置登出縮小窗口；未來 P5 2FA 補強 |
| 建帳號流程改變 | 上線後管理員「不能再設密碼」，工作習慣改變 | 上線時知會管理員；邀請可重發，無不可逆風險 |
| 既有帳號過渡 | 現有帳號全部已有密碼，不受影響；**不會**強制任何人重設 | 無需遷移動作，migration 只加新表/欄位 |
| 寄信基礎建設第一次建 | SMTP 設定錯誤會讓建帳號流程卡住（信寄不出去） | 寄信失敗要有明確錯誤與 audit，且建帳號本身成功、信可重發，不讓交易整包回滾 |
