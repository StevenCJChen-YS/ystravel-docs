# Example Mapping: auth-role-management（Phase 1）

> 2026-07-14 與 Steven 對談完成。8 個問題全數拍板（含 PRD §9 的 5 題＋設計時追加 3 題），規則 6 條。

## Rule 1｜建立角色：code 手填、格式驗證、全域唯一

- ✅ 填 code（`^[A-Z0-9_]+$`）＋名稱＋系統範圍（全系統 or 指定系統；UI 用詞 2026-07-15 由「平台共用」改「全系統」）→ 成功，`isSystem=false`
- ✅ 不勾任何權限的「空角色」允許存在（先建殼、後配權限）
- ❌ code 重複 → 409 明確錯誤
- ❌ code 格式不符（小寫/空白/中文）→ 400
- ❌ 指定不存在的系統代碼 → 400

## Rule 2｜code 與系統範圍建立後不可變

- ❌ 編輯時帶不同 code → 400（前端欄位 disabled，後端仍擋——防 client bug）
- ❌ 編輯時改系統範圍 → 不提供（前端 disabled；改範圍語義上＝另一個角色，需要就建新角色改派）

## Rule 3｜刪除防呆

- ✅ 刪 userCount=0 的一般角色 → 成功（RolePermission cascade 清除）＋audit 留 code/name 快照
- ❌ 刪仍有使用者掛著的角色 → 409，訊息含人數、要求先改派
- ❌ 刪系統角色（isSystem）→ 400

## Rule 4｜系統角色保護矩陣

> **2026-07-15 收斂**：系統角色只剩 **SUPER_ADMIN**（原 `AUTH_SUPER_ADMIN` 改名——全系統角色不掛 AUTH_ 前綴，名稱同步「Auth 超級管理者」→「超級管理者」）。其他出廠角色（AUTH_ADMIN／AUTH_HR／CRM_*）降級為**一般角色**＝起手範本，可改可刪，等同管理員自建。原矩陣的「系統角色（非超管）」欄位走入歷史。

| 操作 | 一般角色 | SUPER_ADMIN（唯一系統角色） |
|---|---|---|
| 改名稱/說明 | ✅ | ✅ |
| 改 code / 系統範圍 | ❌ | ❌ |
| 停用 | ✅ | ❌ 400（防全平台鎖死） |
| 刪除 | ✅（有人掛著 409） | ❌ 400 |
| 改權限 | ✅ | ❌ 400 唯讀（guard 本來就寫死全權，勾了也沒意義） |

- ✅ 超管權限頁可進入，唯讀呈現 DB 實況＋說明「系統寫死全權」

## Rule 5｜權限指派整頁

- ✅ 系統(APP) → **資源(RESOURCE)** 分卡（07-14 拍板改資源：AUTH 權限全掛 admin 模組、按模組會全擠一卡）、每權限一勾（中文動作為主、code 小字）、卡片全選/清除＋搜尋＋全部選取/清除、整頁一次儲存
- ✅ 指定系統的角色只顯示/只能勾**自己系統**的權限；全系統角色才看全部系統
- ✅ 儲存＝全量替換語意、後端 diff 套用（只刪被移除、只建新增）、audit 單筆含 added/removed
- ✅ 清空全部權限（空陣列）→ 允許
- ❌ 未知權限碼 → 400 列出缺碼
- ✅ 未儲存變更時離開頁面（站內跳轉/關分頁）→ 警告確認
- ✅ 生效時點＝下次 token refresh（≤15 分鐘，沿用既有決策，非新決定）；儲存 toast 與驗收文件都標明

## Rule 6｜Seed 不回寫（配合 Rule 4「出廠角色可改」）

- ✅ 部署重跑 seed：角色**不存在才建立**；權限只在角色**首次建立**時寫入
- ✅ 管理員在畫面上對出廠角色做的名稱/說明/權限調整，部署後**保留不被還原**
- ❌ 原行為（upsert 蓋名稱說明＋skipDuplicates 補權限）→ 廢除（否則「出廠角色可改」是假的）
- ✅ 2026-07-15 追加：seed 前置一次性收斂（冪等）——`AUTH_SUPER_ADMIN` 存在且 `SUPER_ADMIN` 不存在＝改名；出廠角色 `isSystem: true → false` 降級
