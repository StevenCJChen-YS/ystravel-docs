## 1. 一句話摘要

讓管理員在 AuthPortal 直接**新增／編輯／刪除角色**，並用一個獨立整頁**勾選分配角色的權限**；並可對**單一使用者**做權限微調（在角色之上額外加／減特定權限）。

## 2. 背景與動機

- 現在角色是唯讀的：後端 `/roles` 只有 GET，前端點進去只有一顆「權限組合」detail modal 看看而已。所有角色都是程式碼種進去的系統角色（`isSystem=true`），管理員無法自己開新角色或調權限。
- 要真正「以角色管權限」上線，管理員得能自訂角色（例如「客服主管」「報表檢視員」）並自己勾權限，而不是每次都找工程師改 seed。
- 少數情況一個人需要「比他角色多／少一項權限」的例外——現有資料表 `UserPermissionOverride` 已經支援、後端有效權限計算也已經算進 override，只差 CRUD 端點與畫面。

## 3. 目標使用者

- **內部管理員**（具 `AUTH.ROLE.MANAGE` 權限者，通常是 IT／超管）。
- 使用頻率：低頻但關鍵——建立/調整角色是偶爾為之，但一旦錯了影響面大（權限過大＝資安風險、權限不足＝有人被鎖在外面）。

## 4. User Story

- 作為**管理員**，我希望能新增一個角色、選它屬於哪個系統、幫它勾好權限，以便把一組權限打包給一群人用，不用每次找工程師改。
- 作為**管理員**，我希望能在一個清楚的整頁上看到「這個角色有哪些權限」並直接勾選調整，以便快速維護。
- 作為**管理員**，我希望能對某個人做權限微調（多給或收回一項），以便處理角色不完全符合的例外，而不用為一個人開一個新角色。

## 5. Acceptance Criteria（驗收標準）

> 每條都是 Steven 能自己在畫面上操作驗證的。

**Phase 1 — 角色 CRUD ＋ 權限分配**
- [ ] 可以「新增角色」：選系統範圍（平台共用 or 指定系統）、填名稱／代碼／說明，建立後出現在角色列表。
- [ ] 可以「編輯角色」：改名稱、說明、啟用/停用狀態（代碼建立後不可改）。
- [ ] 可以「刪除角色」：有防呆（見待釐清；系統角色與「還有使用者掛著」的角色不能隨便刪）。
- [ ] 從角色列表點進「角色權限設定」**獨立整頁**：權限按「系統 → 模組」分區、每個權限一個勾選、卡片可全選/清除、整頁一次儲存。
- [ ] 儲存後，該角色底下的使用者「有效權限」立即反映（可用一個測試帳號驗證）。
- [ ] 系統角色（`isSystem`，如 super_admin）受保護：不可刪、代碼不可改（權限是否可改見待釐清）。

**Phase 2 — 單一使用者權限微調（override）**
- [ ] 在使用者頁可以對某人「額外加一項權限」或「收回一項權限」（在他角色給的權限之上）。
- [ ] 微調後，該使用者的有效權限跟著變（角色權限聯集 → 套用個人 ALLOW/DENY）。
- [ ] 可以清除微調、恢復成「只吃角色權限」。

## 6. UI / UX 重點

- **需要的畫面 / 狀態**：
  - 角色列表頁（沿用現有，加上新增/編輯/刪除動作與進入權限設定的入口）——**取代現有唯讀 detail modal**。
  - 「角色權限設定」獨立整頁（參考 Steven 提供的他系統截圖：分類 → 模組卡 → 操作勾選、批次全選/清除、儲存）。對應 AuthPortal 的權限結構＝系統(APP) → 模組(MODULE) → 每個權限一格勾選(ACTION)。
  - 建立/編輯角色的 modal（沿用 `FormModal`）。
  - 使用者頁的權限微調區（Phase 2）。
- **特別注意的互動/例外**：權限多時的分區與捲動；儲存前的「未存變更」提示；系統角色的鎖定態（唯讀樣式＋提示）；空狀態（該系統下沒有權限時）；儲存失敗 toast。
- **可沿用的既有元件**：`FormModal`、`ToolbarButton`、`EmptyState`、`useConfirm`、`TableCard`／`PageHeader`、深色表頭 UTable；權限勾選卡片為新元件（可抽 `PermissionMatrix` 之類共用）。設計走 `docs/design-system/foundation.md`。

## 7. 資料分級與風險（必填 — 依 SECURITY_AND_ISO27001_BASELINE.md）

| 欄位 | 內容 |
|---|---|
| 涉及的資料類型 | 角色定義、權限代碼、角色↔權限對應、使用者↔角色/權限覆寫對應。**不含**客戶個資／金流。 |
| 資料分級 | **Internal**（權限治理資料，非客戶 Restricted 資料，但錯誤會放大成資安事件） |
| 誰可以看 | 具 `AUTH.ROLE.MANAGE`（角色）／`AUTH.PERMISSION.READ`（權限目錄）者 |
| 誰可以匯出 | 不提供匯出 |
| 誰可以編輯 | 具 `AUTH.ROLE.MANAGE` 者；涉及 super_admin 角色／系統角色時再加限制（見待釐清） |
| 是否涉及 AI | 否 |
| 若涉及 AI，AI 會接觸哪些欄位 | — |
| 風險摘要 | 最壞情況：①權限給太多＝提權（一般帳號拿到管理權）②系統角色被誤刪/誤改＝所有人被鎖、平台癱瘓 ③override DENY 誤用＝把某人鎖在外面。**故需：系統角色保護、刪除防呆、最後一位超管防呆、全程 audit。** |

## 8. 不做的事（Out of Scope）

- **不做「權限目錄」本身的 CRUD**：權限代碼（`AUTH.USER.READ` 這種）是程式定義（`permission-catalog.ts`），這次不在畫面上新增/改權限，只做「把既有權限分配給角色/使用者」。
- 不做角色的順序（order）管理、批次匯入角色。
- 不做「角色繼承／角色群組」等進階建模。
- 不動 SSO 換碼、token 結構（有效權限計算已存在，沿用）。
