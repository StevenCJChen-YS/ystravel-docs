## 1. 一句話摘要

在 CRM 的「ERP 匯入」頁面上傳 ERP 匯出的 Excel,系統把每一列存成原始紀錄,並依已知規則標記哪些列不該成為正式客戶資料。

## 2. 背景與動機

CRM 需要把 ERP 匯出的旅客/訂單 Excel(`docs/analysis/CUSTOMER_EXCEL_FIRST_PASS_ANALYSIS.md` 分析過的那份,約 10 萬列、40 欄)變成 CRM 的客戶與訂單資料。但原始 Excel 混雜了不該變成客戶的資料(導遊、領隊、內部帳號等),必須先清洗與標記,才能進到下一步的「客戶候選比對」(下一個功能,這次不做)。

目前 `Ystravel-CRM-Backend` 只有匯入批次/原始列的**讀取** API(`imports.controller.ts`),完全沒有「上傳並寫入」的功能;`ImportsView.vue` 也已經有一個「匯入 Excel」按鈕,但還沒接任何動作。

## 3. 目標使用者

CRM 內部使用者,具備 `CRM.IMPORT.MANAGE` 權限(業務主管或系統管理員),頻率:每次要匯入一批新的 ERP 匯出資料時使用,預估初期每週數次。

## 4. User Story

作為具備匯入權限的 CRM 使用者
我希望上傳 ERP 匯出的 Excel,系統自動存檔並標記哪些列有問題
以便後續建立客戶資料時,已知不該收錄的資料不會被誤用

## 5. Acceptance Criteria

- [ ] 在「ERP 匯入」頁面點擊「匯入 Excel」可以選擇檔案並上傳
- [ ] 上傳後,畫面上的匯入批次列表會出現這次上傳的紀錄,狀態最終顯示為「已清洗」(NORMALIZED)
- [ ] 點進批次可以看到每一列原始資料,以及是否被標記排除、排除原因
- [ ] 符合下列任一規則的列會被標記排除,但**不會被刪除**(原始資料仍保留):
  - 身份證字號為空
  - 身分為「導遊」或「領隊」
  - 警示訊息有值
  - 來源帳號為「永信旅行社」、「永信旅行社股份有限公司」或「永信福委會」
  - 護照號碼為「未建旅客檔」
- [ ] 上傳一個內容有誤的檔案(例如非 Excel、缺少必要欄位)會出現清楚的錯誤訊息,不會讓系統壞掉

## 6. UI / UX 重點

- 沿用 `ImportsView.vue` 既有的「匯入 Excel」按鈕(目前是空按鈕),點擊後跳出檔案選擇。
- 上傳中要有 loading 狀態,上傳完成後列表要自動刷新。
- 上傳失敗要有清楚的錯誤訊息(不只是「發生錯誤」)。
- 批次詳情頁(如果還沒有,先不用另外做,列表看得到狀態與列數即可,詳情頁面留給下一個功能做)。

## 7. 資料分級與風險(必填)

| 欄位 | 內容 |
|---|---|
| 涉及的資料類型 | 旅客/客戶姓名、身分證字號、護照號碼、電話、Email、地址、訂單金額 |
| 資料分級 | **Restricted**(依 `SECURITY_AND_ISO27001_BASELINE.md` §3.1) |
| 誰可以看 | 具備 `CRM.IMPORT.MANAGE` 權限者 |
| 誰可以匯出 | 這次不做匯出功能 |
| 誰可以編輯 | 這次只做上傳與清洗標記,不開放編輯原始列 |
| 是否涉及 AI | 否(這次的清洗規則是明確規則比對,不是 AI 判斷) |
| 風險摘要 | 上傳檔案含大量真實旅客個資;若規則寫錯導致不該排除的資料被誤標,或該排除的資料沒被標記,會影響後續客戶資料品質。原始檔案暫不落地保存(只解析後存資料庫),降低檔案外流風險。 |

## 8. 不做的事(Out of Scope)

- 不做客戶候選比對、客戶合併(下一個功能)。
- 不做「同業 / 直客」分類(需要另一份同業名單,尚未提供)。
- 不做員工身分證黑名單比對(尚未提供名單)。
- 不決定「整筆列排除」還是「整張訂單排除」的最終邏輯 —— 這次只在**列層級**標記排除原因,訂單層級的處理策略留給候選比對功能決定。
- 不做批次刪除、重新上傳覆蓋等進階批次管理。
