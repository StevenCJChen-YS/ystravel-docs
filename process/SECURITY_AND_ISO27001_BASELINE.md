# Security And ISO 27001 Baseline

## 1. 目的

本文件定義之後開發 SSO、CRM、EIP 整合、Excel 匯入、旅客/訂單分析、AI 輔助流程時，最低限度必須遵守的資安與文件規範。

本文件的目標不是宣稱「現在已通過 ISO 認證」，而是讓後續設計、開發、測試、部署與營運，都朝 `ISO/IEC 27001:2022` 導向靠齊。

## 2. 重要觀念

### 2.1 ISO 27001 不是單一功能或單一系統的勾選清單

`ISO/IEC 27001` 是資訊安全管理系統（ISMS）的要求標準，重點是：

- 風險評估
- 控制措施選擇
- 文件化
- 角色責任
- 稽核
- 持續改善

所以：

- 不是把某個 API 做完就叫符合 ISO 27001
- 不是只有程式碼安全就夠
- 也包含人員流程、權限治理、供應商、備份、事件處理、稽核紀錄

### 2.2 對本專案的實際意思

對你目前要做的事，`符合 ISO 27001 導向` 的意思是：

- 敏感資料先分類
- 存取權限要最小化
- 有風險評估與例外管理
- 文件必須記錄資料流與控制點
- 開發流程必須能追溯誰改了什麼
- AI 使用必須有邊界

## 3. 資料分級

### 3.1 Restricted

以下資料預設列為 `Restricted`：

- 身分證字號
- 護照號碼
- 台胞證號
- 生日
- 手機
- Email
- 地址
- 旅客名單
- 訂單明細
- 付款/金額資料
- 任何可直接或間接識別旅客身份的欄位組合

規則：

- 不可任意匯出
- 不可任意用於測試資料
- 不可上傳到未核准的外部 AI 服務
- 不可在多人共用群組、聊天工具、郵件中隨意散布

### 3.2 Internal

- 欄位規格
- 去識別化統計報表
- 權限模型
- 系統設計文件

### 3.3 Public

- 對外公開的公司介紹
- 不含敏感資訊的公開文件

## 4. Excel 與 ERP 原始資料處理規則

### 4.1 原始 Excel 不必強制去識別化

若業務目的是要正確辨識旅客、訂單、重複身份、聯絡資訊與關聯資料，原始 Excel 在分析階段通常不能完全去識別化。

這是合理的，但前提是：

- 分析環境受控
- 存取人員最小化
- 不把原始個資丟到未核准的外部服務
- 有保留與刪除規則
- 有操作紀錄與輸出管制

### 4.2 原始資料處理原則

- 原始 Excel 僅能進入受控 workspace 或受控匯入區。
- 原始檔不得直接散落在多個個人資料夾與聊天附件中。
- 匯入後保留 `raw layer`，但只開放少數角色查看。
- 正式分析畫面與日常 CRM 畫面應優先使用清洗後與遮罩後資料。

### 4.3 測試與開發禁止事項

- 不得把真實旅客個資直接寫進前端假資料。
- 不得把真實 Excel 放進 repo。
- 不得把真實個資直接放進單元測試 fixture。
- 不得把 raw data 直接交給外部 AI 作摘要或分類，除非經正式核准。

## 5. AI 使用規則

### 5.1 第一階段原則

第一階段 AI 不處理以下高風險決策：

- 正式客戶合併
- 正式身份唯一判定
- 高風險權限核准
- 未經審查的個資外送

### 5.2 可接受的 AI 輔助範圍

在風險可控前提下，AI 可用於：

- 欄位命名建議
- 地址格式解析建議
- 國家/通路分類建議
- 髒資料標註建議
- 可疑重複客戶的候選排序
- 報表摘要草稿

### 5.3 AI 使用前必填紀錄

若某功能要使用 AI，文件中至少要補：

- 使用目的
- 會接觸哪些資料欄位
- 是否包含 Restricted 資料
- 模型位置：本地 / 內網 / 雲端
- 是否可逆識別
- 人工審核點
- 失敗與誤判風險

## 6. 功能文件必填資安欄位

之後每個重要功能的 `PRD / SA / SD / API Spec` 至少要補以下欄位：

### 6.1 PRD 必填

- 功能目的
- 涉及資料類型
- 資料分級
- 使用者角色
- 誰可以看 / 匯出 / 編輯
- 是否涉及 AI
- 風險摘要

### 6.2 SA 必填

- 資料流
- 系統邊界
- 敏感資料流向
- 風險點
- 人工審核點
- 與外部系統交互方式

### 6.3 SD 必填

- 認證與授權方式
- 欄位加密 / 雜湊 / 遮罩策略
- Audit log 設計
- 備份 / 還原策略
- 錯誤處理與告警
- 權限控管與角色矩陣
- 測試資料策略

### 6.4 API Spec 必填

- 權限需求
- 輸入輸出敏感欄位
- 是否支援批次匯出
- 速率限制或匯出限制
- 日誌紀錄要求

## 7. 最低技術控制要求

### 7.1 存取控制

- 全系統走 AuthService SSO
- 角色與權限分離
- 最小權限原則
- 高敏感操作需可追溯

### 7.2 敏感欄位保護

- 身份證件值優先採 `hash + masked value`
- 憑證、金鑰、密碼不得寫死在程式碼或文件
- 測試環境避免使用真實 PII

### 7.3 日誌與稽核

以下動作要記錄 audit log：

- 登入 / 登出
- 匯入 Excel
- 匯出資料
- 權限異動
- 客戶合併核准
- 人工覆核結果

### 7.4 備份與刪除

- 匯入批次要有保存期限
- raw file 要有刪除與封存規則
- 備份資料也屬於敏感資料範圍

### 7.5 開發與部署

- Repo 不放真實個資
- 環境變數與 secrets 分離管理
- Production / staging / local 權限分離
- 重大變更要有 review 與回滾方案

## 8. 與 ISO 27001 導向最相關的執行項

若你之後真的要朝認證前進，最少要補這些治理產物：

- 資訊資產清冊
- 風險登錄表
- 資料分級政策
- 存取控制政策
- 事件通報流程
- 備份與還原流程
- 第三方與外部 AI 使用規範
- 稽核紀錄與管理審查紀錄
- 適用性聲明（Statement of Applicability, SoA）

## 9. 第一階段強制規則

從現在開始，以下視為強制：

1. 真實旅客/訂單 Excel 不進 git repo。
2. 真實個資不上未核准外部 AI。
3. 所有 CRM 匯入功能都要先有 raw layer，再談正式 customers。
4. 所有匯入、匯出、合併、權限異動都要可稽核。
5. 每個重要功能文件都必須補資料分級與權限說明。
